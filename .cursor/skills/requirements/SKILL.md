# Requirements

Use this skill to capture unknowns, close gaps, and produce a **Functional Specification Document (FSD)** that drives design and implementation. It is invoked by the spec-driven-development skill when no usable spec exists.

## Core Rules

- Produce a **gaps questionnaire** via the **Gap Manager Portal** (globally managed) before writing the full FSD when doing full requirement analysis.
- Each **project** owns its **own JSON** questionnaire file; the portal maintains a global **registry** of project JSON locations.
- Treat the **per-project JSON** as the source of truth; generate the Markdown questionnaire + answers from that JSON for review and traceability.
- Use the **filled questionnaire** as the primary input to the FSD so requirements are explicit and traceable.
- For small tasks, the questionnaire may be a short checklist; the spec must still be explicit enough that another engineer could tell whether the work is done.
- When the project exposes or consumes APIs, align integration and data contract details with the OpenAPI spec produced in the design phase.
- **Hard enforcement (manual PR operations):** This skill MUST NOT create, open, update, comment on, approve, or merge pull requests. Any PR/merge action is **manual developer work**; this skill only defines requirements artifacts and required evidence.
- Maintain an **OpenSpec spec library** in the project so behavior intent persists beyond the FSD:
  - Source-of-truth specs: `<projectDir>/openspec/specs/**/spec.md`
  - Specs MUST remain behavior-first (requirements + scenarios), not implementation.

## Gaps Questionnaire Workflow

**Project root convention:** `<projectDir>` is the folder directly under the repo root (`Presales/<ProjectName>`). The **project name** is `<ProjectName>`.

1. **Ensure the Gap Manager Portal is running** (auto-start if needed):
   - Run `node system-prompts-skills/requirement-skill/gaps-questionnaire-web/ensure-portal-running.mjs`
   - Policy: always use port **4317**; if something else is on that port, stop it and start the portal.
2. **Create the per-project folder layout** (once per project):
   - Use `init-project-structure.mjs` to create `<projectDir>/Discovery and Design`, `<projectDir>/Docs`, `<projectDir>/codebase`.
3. **Create the per-project gaps JSON** (the original gaps document is JSON-first):
   - Use `init-gap-json.mjs` to create `<projectDir>/Discovery and Design/gap-questionnaire.json` if missing.
4. **Auto-register the project** by JSON path (no manual project name):
   - Call `POST /api/register-from-path` with `{ jsonPath }` where `jsonPath = "<projectDir>/Discovery and Design/gap-questionnaire.json"`
   - The portal infers **project name = `<projectDir>` folder name**
5. **Fill** each section in the portal: answers, owner, confidence (H/M/L), then **Save JSON**.
   - Selecting the project loads questions directly from `<projectDir>/Discovery and Design/gap-questionnaire.json`.
6. **Stage gate (hard stop):** do not proceed to FSD/architecture/coding until gaps are finalized by setting **Status = Complete** (confirmation required; portal saves + finalizes).
7. (Optional but recommended) **Generate Markdown** outputs into `<projectDir>/Docs/` and auto-register the artifact:
   - Run `generate-gap-md.mjs --in "<projectDir>/Discovery and Design/gap-questionnaire.json" --projectDir "<projectDir>"`
   - Call `POST /api/register-artifact` with `{ name, jsonPath, questionnaireMdPath, answersMdPath }` pointing to `<projectDir>/Docs/...`
8. **Generate an assumptions document for any remaining gaps** (required if anything is unresolved):
   - Run `generate-assumptions-md.mjs --in "<projectDir>/Discovery and Design/gap-questionnaire.json" --projectDir "<projectDir>"`
   - This writes `<projectDir>/Docs/SDD_Assumptions.md` listing unresolved items and how they are handled (assume/defer/block).
9. **Use** the filled questionnaire (JSON + generated Markdown) to create/update the FSD in `<projectDir>/Docs/`, keeping traceability.

Questionnaire sections: Functional gaps; Data and domain gaps; Integration and API gaps; Non-functional and constraints; Scope and out-of-scope; Open decisions; Traceability to FSD (after FSD creation).

## FSD Content Requirements

A usable spec must define:

- **Executive summary** — 2–4 sentences: what is being built, why, who it serves
- **Background & problem statement** — current state, pain or gap, how users cope today
- **Goals & success criteria** — specific success metrics; explicit out-of-scope
- **Stakeholders & user personas** — roles, personas, key scenarios
- **Functional requirements** — by Epic: User Stories with Acceptance Criteria (Given/When/Then)
- **Non-functional requirements** — performance, security, reliability, scalability, compliance, usability
- **System & data architecture (high-level)** — key entities, integration points (with endpoints, auth, sync, error handling). For APIs owned by the project, align with the **OpenAPI specification**
- **User flows** — 2–3 key end-to-end workflows; note error states
- **Assumptions** — list explicitly for validation
- **Constraints** — technical, budget/time, regulatory, organizational
- **Open questions** — decision or information needed, owner, due
- **Out of scope (parking lot)** — deferred items and reason
- **Glossary** — domain-specific terms

## OpenSpec alignment

After the FSD is drafted (or updated), extract the durable behavior contract into OpenSpec specs under `<projectDir>/openspec/specs/`:

- Each domain spec includes `## Purpose` and `## Requirements`.
- Requirements use RFC 2119 keywords and include Given/When/Then scenarios.

The FSD remains the design-phase narrative; OpenSpec becomes the long-lived, behavior-first source of truth.

## Section Guidance (from FSD template)

- **Keep** Background when context isn’t obvious; Personas when multiple user types; Data architecture for greenfield or complex data; Integration points when external systems are involved; User flows for complex workflows; Glossary when domain terminology exists.
- **Drop** sections that have no content for the task; calibrate detail to input (one-liner → 1–2 page spec; meeting transcript → 3–6 page spec with full epics).

## Failure Modes To Avoid

- Writing the FSD without filling the gaps questionnaire first (for non-trivial work).
- Vague acceptance criteria that cannot be tested or validated.
- Integration points described at a high level without endpoints, auth, and error handling.
- Omitting explicit out-of-scope and assumptions.

## Handoff

When handing off to architecture or implementation: state spec status (draft/approved); list FSD section IDs or requirement IDs that are approved; note open questions and owners. This skill succeeds when the FSD is the single source of truth for what will be built and validated.
